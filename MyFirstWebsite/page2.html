Theory

How Git Handles Data

After initiating the git repository, Git takes a "snapshot" of the current state of the project whenever the user makes changes to the repository.
Git basically takes a picture of what all the files look like at that moment and stores a reference to that snapshot.

Local operations

Almost every Git operation can be done on local machine, since most operations in Git need only local files and resources to operate.
There is no need for an external machine or network for editing and committing files to a local repository. Internet is needed only when accessing remote repositories like GitHub. And because the whole history is on the local machine, the operations are almost instantaneous.

Integrity

Everything in Git is checksummed before being stored and then is referred to by that checksum. This way Git repository keeps track of everything inside of it and it is nearly impossible to make changes without Git knowing about it.

Git only adds data

Git actions only add data to Git database, so it is hard to get the system to do anything that is not undoable or to make it erase data in any way. As with any VCS, it is possible to lose or mess up changes that havenâ€™t been committed yet, but after committing a snapshot into Git, it is very difficult to lose, especially if users regularly push their database to another repository.

Committed, Modified, Staged

The files in Git can be categorized in 3 different states: Committed, Modified and Staged.

  • Committed: data is safely stored in database

  • Modified: file has been modified, but not committed to database

  • Staged: file has been marked to be included in next commit

The 3 main sections of Git project are: Git repository, working directory and staging area. The working directory is where user does all the editing work on files. After that, the files are added to the staging area to be committed. When the files are committed, the files from staging area are moved permanently to the repository database.

Basic workflow on Git:
0. (Optional) Clone repository from remote repository.
1. modify files in working directory.
2. selected files to be committed in the staging area.
3. commit files to repository from staging area. Git takes a snapshot of that commit and stores it permanently in the Git directory (repository).
